## 2. 背景与问题

梦境记录通常存在三个问题：

1. 用户醒来后记忆快速消散，零散文本难以长期回顾。
2. 传统日记只能保存原文，缺少情绪、压力、人物、地点、意象等结构化整理能力。
3. 梦境天然具有画面感，但纯文本记录难以还原梦境氛围，也难以激发用户持续记录。

Dream Reel 希望把一次梦境记录转化为可保存、可视觉化、可回顾的私人档案。产品围绕“记录梦境 - AI 解析 - AI 追问 - 图像生成 - 档案回顾 - 周期复盘”建立核心链路。

## 3. 产品目标

### 3.1 用户目标

- 快速记录刚醒来时的梦境片段。
- 借助 AI 自动整理梦境中的情绪、压力、人物、地点和意象。
- 通过追问补充遗漏细节，让梦境记录更完整。
- 根据梦境内容生成视觉图像，增强回忆与保存价值。
- 在档案中查看历史梦境，发现重复出现的情绪、人物、地点和符号。

### 3.2 业务目标

- 完成一个可上线、可注册、可持续使用的 AI 日记产品。
- 通过免费额度和 Plus 额度控制 AI 成本。
- 支持中英双语，扩大潜在用户范围。
- 为后续用户增长、付费转化和梦境数据分析能力打基础。

### 3.3 技术目标

- 使用 PostgreSQL 持久化用户、梦境、订阅和用量数据。
- 使用 Auth.js 实现邮箱密码注册登录与用户级数据隔离。
- 使用 OpenAI 完成梦境分析、对话追问、标题生成和图像生成。
- 对梦境原文进行加密存储，降低隐私风险。
- 所有 API 入参使用 Zod 校验，避免非法数据进入数据库。

## 5. 用户场景

### 场景 1：醒来后快速记录梦境

用户醒来后记得一些片段，希望在忘记前快速输入。系统需要允许用户直接输入文本，补充睡眠质量、睡前饮食、睡前活动等信息，并保存为一条梦境记录。

### 场景 2：用户希望 AI 帮忙整理梦境

用户输入一段散乱梦境后，点击 AI 分析。系统需要返回标题、情绪、压力分数、人物、地点、意象、睡眠洞察、追问问题和图像提示词，并自动填充到表单。

### 场景 3：用户想补充更多细节

AI 分析后生成 2-3 个追问问题。用户可以回答其中任意问题，系统将问答追加到梦境文本中，并支持再次分析。

### 场景 4：用户想把梦境变成图像

用户确认图像提示词后点击生成图像。系统调用图像生成接口，返回梦境图像，并允许用户保存到当前记录。

### 场景 5：用户回顾历史梦境

用户进入档案页，通过日历和最近列表查看历史梦境，编辑记录、重新分析、重新生成图像，并查看本周统计。

### 场景 6：用户额度不足

用户当月 AI 分析或图像生成额度用尽时，系统需要阻止继续消耗接口，提示升级 Plus 或下月继续使用。

## 7. 核心流程

### 7.1 新用户流程

1. 用户进入首页。
2. 点击开始记录。
3. 未登录用户跳转至登录页。
4. 用户注册或登录。
5. 进入 Journal 页面创建第一条梦境。

### 7.2 梦境记录流程

1. 用户输入梦境文本。
2. 用户选择输入模式：文本或语音。
3. 用户可选填写入睡时间、醒来时间、睡眠质量、睡前饮食、睡前活动。
4. 用户点击 AI 分析。
5. 系统返回结构化字段并填充表单。
6. 用户可回答 AI 追问并重新分析。
7. 用户可生成梦境图像。
8. 用户保存梦境。
9. 系统扣减梦境记录、AI 分析、图像生成对应额度。

### 7.3 档案回顾流程

1. 用户进入 Archive 页面。
2. 系统加载当前用户的梦境列表和本周统计。
3. 用户通过日历或最近列表选择梦境。
4. 用户打开编辑弹窗。
5. 用户修改字段、重新分析或重新生成图像。
6. 用户保存更新或删除记录。

## 11. 非功能需求

### 11.1 性能

- 常规页面首屏加载目标：3 秒内可交互。
- AI 分析接口超时时间：60 秒。
- 图像生成接口超时时间：170 秒。
- 数据库连接池在 serverless 环境中限制单实例连接数，避免连接耗尽。

### 11.2 安全与隐私

- 未登录用户不能创建、编辑、删除梦境。
- 梦境查询、编辑、删除必须按 user_id 隔离。
- 梦境原文和整理文本存储前加密。
- API Key、Auth Secret、加密密钥只能通过环境变量读取。
- 所有写入接口必须进行 Zod 校验。
- 支付 Webhook 需要记录事件，避免重复处理风险。

### 11.3 稳定性

- AI 服务失败时返回可理解错误，不保存半成品数据。
- 已扣额度的失败操作需要退回额度。
- 数据库 schema 迁移需要幂等，支持冷启动自动补齐字段。
- AI 输出 JSON 解析失败时需要有兜底错误处理。

### 11.4 可维护性

- 数据库访问集中在 `src/lib/dreams.ts`、`src/lib/db.ts`。
- AI Prompt 与解析逻辑应尽量模块化，避免散落在多个 UI 组件中。
- 表单字段、API schema、数据库字段命名需要保持一致。
- 中英文本集中维护，减少硬编码。

## 12. 验收标准

### 12.1 核心链路验收

- 新用户可以注册、登录并进入 Journal。
- 登录用户可以创建一条包含梦境文本和睡眠字段的记录。
- 点击 AI 分析后，页面能填充标题、情绪、压力分数、人物、地点、意象、睡眠洞察、追问问题和 visualBrief。
- 用户回答追问后，问答内容能追加到梦境文本。
- 用户可以生成图片，并将图片保存到记录中。
- 用户可以在 Archive 页面查看、编辑、删除该记录。
- Weekly Recap 能正确展示本周梦境聚合结果。

### 12.2 权限与额度验收

- 未登录用户调用创建、更新、删除、分析、图像生成接口时会被拒绝。
- 用户不能通过接口修改其他用户的梦境。
- Free 用户超过月度额度后，继续分析或生成图像会收到额度不足提示。
- 图像生成失败或超时时，本次图像额度会退回。

### 12.3 数据验收

- raw_text 和 clean_text 在数据库中不是明文。
- 读取梦境时，前端展示的是解密后的可读文本。
- stress_score 只能为 1-5 或 null。
- people、locations、symbols 为空时返回空数组。
- captured_at 缺省时使用当前时间。

### 12.4 双语验收

- 用户切换中文 / English 后，主要页面文案同步切换。
- AI 分析和标题生成跟随 lang 返回对应语言。
- 页面跳转后语言选择保持不变。

## 14. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AI 输出格式不稳定 | 前端无法填充字段 | 使用 response schema、Zod 校验、兜底解析和明确错误提示 |
| 图像生成耗时长 | 用户等待体验差 | 设置超时、加载态、失败退额、提示稍后重试 |
| AI 成本增长 | 运营成本不可控 | 用量系统、免费额度、Plus 额度、失败退额 |
| 梦境内容隐私敏感 | 用户信任受损 | 登录隔离、文本加密、减少日志输出敏感内容 |
| 结构化字段过度主观 | 用户不认同分析结果 | 支持用户编辑字段，Prompt 中强调抽取而非诊断 |
| 睡眠分析被误解为医疗建议 | 合规风险 | 文案避免诊断表达，定位为个人记录和情绪觉察工具 |
